Shared platform basis for VB and EVA

The meeting confirmed an important product idea: Virtual Buddy and EVA can share the same underlying platform base. The first tier can be almost identical technically; the difference is whether the assistant is pointed inward to employees and knowledge sources, or outward to customers and backend customer journeys.

Common base
  • Frontend channel, application runtime, knowledge ingestion, RAG, LLM gateway, testing, and monitoring
  • Reusable components can be switched on/off by package, add-on, or customer configuration
  • Start packages should focus on fast pilot value before custom agents and integrations are added
Product split
  • Virtual Buddy points the platform at internal knowledge, employees, policies, and process documentation
  • EVA points the platform at customer journeys, external channels, backend context, and action execution
  • Advanced tiers can converge again through agents, custom workflows, integrations, SARA, and AI testing

Data preparation vs. runtime flow

The transcript separates two different technical phases. First, data must be prepared and indexed. Only after that does the runtime chat/RAG flow answer user questions from the prepared knowledge layer.

Preparation path A: File upload
  • Customer uploads files, HTML export, ZIP package, or documentation bundle into controlled storage
  • Pipeline parses, chunks, versions, and stores metadata for each document version
  • Embedding model creates vectors and writes chunks into vector search / AI Search
Preparation path B: Source connector
  • Customer authorizes Confluence, SharePoint, web, file storage, or another source through a token/connector setup
  • Connector runs scheduled refresh and indexes changed content, for example once per day
  • Future product need: admin UI for connector setup, parsing rules, refresh cadence, and source scope
Runtime query flow
  • User asks a question through chat or another frontend; conversation history is stored in query/audit DB
  • Question is embedded and nearest chunks are retrieved from vector search
  • Retrieved chunks are added to the prompt and passed through backend to the LLM gateway
Answer and control
  • LLM returns an answer with references/citations when supported by the source content
  • SARA/testing layer validates answer quality, policy compliance, and regression behavior
  • Storage is mainly preparation-time infrastructure; runtime primarily uses query DB, vector DB, backend, and LLM gateway

Three deployment models

1. O2-hosted SaaS
  • VB/EVA application, storage, LLM gateway, SARA, and monitoring run in O2 environment
  • Customer data is accessed through secure connectors, APIs, VPN, Private Link, or controlled import
  • Best for fast start, repeatable delivery, and strongest O2 control over updates and usage billing
2. Customer-cloud managed instance
  • Runtime runs in customer cloud as a managed O2 product, typically via containers and controlled release process
  • Customer does not receive source code, build pipeline, or rights to modify the solution
  • Best for regulated clients that require data and runtime close to their cloud boundary
3. Hybrid control plane + customer data/action plane
  • O2 keeps product control plane: orchestration, LLM gateway, SARA, metering, updates, and monitoring
  • Customer hosts data/action plane: connectors, indexer, vector store, API proxy, or action adapters
  • Best target model for enterprise customers that require data control without losing O2 product governance
Architecture decision
  • Default for speed: O2-hosted SaaS
  • Default for enterprise: Hybrid control plane + customer data/action plane
  • Exception model: Customer-cloud managed instance for strict compliance cases

Virtual Buddy specifics

Virtual Buddy is primarily a knowledge-access product. The main architecture question is where documents, embeddings, query history, and source connectors live, and how O2 keeps LLM usage and updates under control.

Core infrastructure
  • User FE for employee queries
  • Document storage and source connectors
  • Query/audit database and vector index
  • VB application cloud runtime
  • Lite LLM PTU model with pay-as-you-go fallback models
  • User policy layer controlling who can access which buddy and which knowledge scope
Main workshop decisions
  • Are documents copied, synchronized, indexed in place, or queried on demand?
  • Who owns embeddings, query logs, retention, and deletion?
  • Does the customer use O2 LLM gateway or their own model subscription?
  • How is usage metered for subscription, token resale, and managed operations?
  • Which add-ons are standard: connectors, testing, SARA, admin UI, policy management?

EVA specifics

EVA uses the same deployment models, but is more sensitive because it can serve external customers, verify identity, call backend systems, and execute actions. The key question is not only where EVA runs, but where customer actions are authorized and executed.

Additional EVA layers
  • Identity verification and authorization/entitlement engine
  • Customer context resolver and journey state management
  • Action execution layer for backend systems
  • Human handoff, audit log, error handling, and compensation logic
  • SLA and incident process for customer-facing impact
Use-case risk classes
  • Informational: order status, billing info, ticket status
  • Guided workflow: data collection, ticket creation, change request preparation
  • Transactional/action: service changes, orders, provisioning, customer-impacting actions

Commercial and IP control rules

  • Frontend and backend should never call models directly; all model calls should go through O2 LLM Gateway.
  • O2 should preserve source-code ownership, build pipeline, release management, and product roadmap control.
  • Even in customer-cloud deployment, O2 should keep license checks, usage metering, managed updates, and support telemetry.
  • Token usage should be billed through a clear model: platform subscription, LLM/token usage, connectors, operations, and change requests.
  • Customer-cloud does not mean software handover; it should be positioned as a managed private instance.
  • O2-owned LLM/API keys should remain in backend or gateway configuration, not visible to the customer frontend or tenant users.
  • If customers use their own model subscription, the commercial model must explicitly separate platform fee from customer-paid model consumption.
