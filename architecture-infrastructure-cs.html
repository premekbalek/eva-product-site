Příprava dat vs. provozní flow

Přepis odděluje dvě technické fáze. Nejdřív se data připraví a zaindexují. Teprve potom runtime chat/RAG flow odpovídá na dotazy z připravené znalostní vrstvy.

Příprava A: Upload souborů
  • Zákazník nahraje soubory, HTML export, ZIP balík nebo dokumentační bundle do řízeného storage
  • Pipeline dokumenty parsuje, chunkuje, verzuje a ukládá metadata ke každé verzi dokumentu
  • Embedding model vytvoří vektory a zapíše chunky do vector search / AI Search
Příprava B: Zdrojový konektor
  • Zákazník autorizuje Confluence, SharePoint, web, file storage nebo jiný zdroj přes token/konektor
  • Konektor běží v naplánovaném refreshi a indexuje změněný obsah, například jednou denně
  • Budoucí produktová potřeba: admin UI pro setup konektorů, parsing rules, refresh cadence a scope zdrojů
Runtime query flow
  • Uživatel položí dotaz přes chat nebo jiný frontend; historie konverzace se ukládá do query/audit DB
  • Dotaz se převede na embedding a z vector search se vytáhnou nejbližší chunky
  • Vybrané chunky se přimíchají do promptu a přes backend jdou do LLM gateway
Odpověď a kontrola
  • LLM vrátí odpověď s referencemi/citacemi, pokud je podporuje zdrojový obsah
  • SARA/testing vrstva validuje kvalitu odpovědi, policy compliance a regresní chování
  • Storage je hlavně přípravná infrastruktura; runtime používá primárně query DB, vector DB, backend a LLM gateway

Tři deployment modely

1. O2-hosted SaaS
  • VB/EVA aplikace, storage, LLM gateway, SARA a monitoring běží v prostředí O2
  • Zákaznická data se připojují přes secure konektory, API, VPN, Private Link nebo řízený import
  • Nejlepší pro rychlý start, opakovatelnou delivery a nejsilnější kontrolu O2 nad updaty a billingem
2. Customer-cloud managed instance
  • Runtime běží v cloudu zákazníka jako managed O2 produkt, typicky přes kontejnery a řízený release proces
  • Zákazník nedostává zdrojový kód, build pipeline ani právo řešení upravovat
  • Vhodné pro regulované klienty, kteří vyžadují data a runtime blízko své cloudové hranice
3. Hybrid control plane + customer data/action plane
  • O2 drží product control plane: orchestraci, LLM gateway, SARA, metering, updaty a monitoring
  • Zákazník hostuje data/action plane: konektory, indexer, vector store, API proxy nebo action adaptery
  • Cílový model pro enterprise zákazníky, kteří chtějí kontrolu dat bez ztráty O2 product governance
Architektonické rozhodnutí
  • Default pro rychlost: O2-hosted SaaS v případě VB
  • Default pro enterprise: Hybrid control plane + customer data/action plane
  • V případě EVA půjde o customizované řešení
