Architektura a infrastruktura

Tři deployment modely pro Virtual Buddy a EVA s jasnými hranicemi pro data, vlastnictví kódu, LLM spotřebu, provoz a odpovědnosti v cloudu zákazníka.

Referenční komponenty

Uživatelské kanályWeb chat, interní UI, aplikace, voice, API
Produktový frontendVB zaměstnanecké FE nebo EVA zákaznický kanál
Aplikační vrstvaOrchestrace, stav konverzace, tenant pravidla, admin
Znalostní/datová vrstvaDocument storage, query DB, audit DB, vector index, konektory
LLM gatewayLite LLM s PTU, pay-as-you-go fallback modely, routing a metering
Kvalita a provozSARA, monitoring, billing, reporting, support, runbooky

Společný platformní základ pro VB a EVA

Ze schůzky vyplývá důležitá produktová myšlenka: Virtual Buddy a EVA mohou sdílet stejný podvozek. První varianta může být technicky velmi podobná; rozdíl je hlavně v tom, jestli se asistent namíří dovnitř na zaměstnance a znalostní zdroje, nebo ven na zákazníky a zákaznické backendové cesty.

Společný základ
  • Frontend kanál, aplikační runtime, ingestion znalostí, RAG, LLM gateway, testing a monitoring
  • Znovupoužitelné komponenty se zapínají/vypínají podle balíčku, add-onu nebo zákaznické konfigurace
  • Start balíčky mají cílit na rychlou hodnotu pilotu před custom agenty a integracemi
Rozdělení produktů
  • Virtual Buddy míří na interní znalosti, zaměstnance, politiky a procesní dokumentaci
  • EVA míří na zákaznické journeys, externí kanály, backendový kontext a vykonávání akcí
  • Vyšší varianty se mohou znovu sbíhat přes agenty, custom workflows, integrace, SARA a AI testing

Příprava dat vs. provozní flow

Přepis odděluje dvě technické fáze. Nejdřív se data připraví a zaindexují. Teprve potom runtime chat/RAG flow odpovídá na dotazy z připravené znalostní vrstvy.

Příprava A: Upload souborů
  • Zákazník nahraje soubory, HTML export, ZIP balík nebo dokumentační bundle do řízeného storage
  • Pipeline dokumenty parsuje, chunkuje, verzuje a ukládá metadata ke každé verzi dokumentu
  • Embedding model vytvoří vektory a zapíše chunky do vector search / AI Search
Příprava B: Zdrojový konektor
  • Zákazník autorizuje Confluence, SharePoint, web, file storage nebo jiný zdroj přes token/konektor
  • Konektor běží v naplánovaném refreshi a indexuje změněný obsah, například jednou denně
  • Budoucí produktová potřeba: admin UI pro setup konektorů, parsing rules, refresh cadence a scope zdrojů
Runtime query flow
  • Uživatel položí dotaz přes chat nebo jiný frontend; historie konverzace se ukládá do query/audit DB
  • Dotaz se převede na embedding a z vector search se vytáhnou nejbližší chunky
  • Vybrané chunky se přimíchají do promptu a přes backend jdou do LLM gateway
Odpověď a kontrola
  • LLM vrátí odpověď s referencemi/citacemi, pokud je podporuje zdrojový obsah
  • SARA/testing vrstva validuje kvalitu odpovědi, policy compliance a regresní chování
  • Storage je hlavně přípravná infrastruktura; runtime používá primárně query DB, vector DB, backend a LLM gateway

Tři deployment modely

1. O2-hosted SaaS
  • VB/EVA aplikace, storage, LLM gateway, SARA a monitoring běží v prostředí O2
  • Zákaznická data se připojují přes secure konektory, API, VPN, Private Link nebo řízený import
  • Nejlepší pro rychlý start, opakovatelnou delivery a nejsilnější kontrolu O2 nad updaty a billingem
2. Customer-cloud managed instance
  • Runtime běží v cloudu zákazníka jako managed O2 produkt, typicky přes kontejnery a řízený release proces
  • Zákazník nedostává zdrojový kód, build pipeline ani právo řešení upravovat
  • Vhodné pro regulované klienty, kteří vyžadují data a runtime blízko své cloudové hranice
3. Hybrid control plane + customer data/action plane
  • O2 drží product control plane: orchestraci, LLM gateway, SARA, metering, updaty a monitoring
  • Zákazník hostuje data/action plane: konektory, indexer, vector store, API proxy nebo action adaptery
  • Cílový model pro enterprise zákazníky, kteří chtějí kontrolu dat bez ztráty O2 product governance
Architektonické rozhodnutí
  • Default pro rychlost: O2-hosted SaaS
  • Default pro enterprise: Hybrid control plane + customer data/action plane
  • Výjimkový model: Customer-cloud managed instance pro přísné compliance případy

Specifika Virtual Buddy

Virtual Buddy je primárně produkt pro přístup ke znalostem. Hlavní architektonická otázka je, kde žijí dokumenty, embeddings, historie dotazů a zdrojové konektory, a jak O2 udrží pod kontrolou LLM spotřebu a aktualizace.

Core infrastruktura
  • Uživatelské FE pro zaměstnanecké dotazy
  • Document storage a zdrojové konektory
  • Query/audit databáze a vector index
  • VB aplikační cloud runtime
  • Lite LLM PTU model s pay-as-you-go fallback modely
  • User policy vrstva určující, kdo má přístup ke kterému buddymu a znalostnímu scope
Hlavní workshop rozhodnutí
  • Budou dokumenty kopírované, synchronizované, indexované u zdroje nebo dotazované on demand?
  • Kdo vlastní embeddings, query logy, retention a mazání?
  • Používá zákazník O2 LLM gateway nebo vlastní model subscription?
  • Jak se měří usage pro subscription, resale tokenů a managed operations?
  • Které add-ony budou standard: konektory, testing, SARA, admin UI, policy management?

Specifika EVA

EVA používá stejné deployment modely, ale je citlivější, protože obsluhuje externí zákazníky, ověřuje identitu, volá backendové systémy a může provádět akce. Hlavní otázka tedy není jen kde EVA běží, ale kde se autorizují a vykonávají zákaznické akce.

Další EVA vrstvy
  • Identity verification a authorization/entitlement engine
  • Customer context resolver a journey state management
  • Action execution layer pro backend systémy
  • Human handoff, audit log, error handling a compensation logic
  • SLA a incident process pro zákaznický dopad
Rizikové třídy use casů
  • Informational: stav objednávky, faktura, ticket status
  • Guided workflow: sběr dat, založení ticketu, příprava změnového požadavku
  • Transactional/action: změna služby, objednávky, provisioning, zákaznicky dopadové akce

Komerční a IP kontrolní pravidla

  • Frontend ani backend by neměl volat modely přímo; všechna modelová volání mají jít přes O2 LLM Gateway.
  • O2 má držet vlastnictví zdrojového kódu, build pipeline, release management a kontrolu produktové roadmapy.
  • I u customer-cloud deploymentu má O2 držet licenční kontrolu, usage metering, managed updaty a support telemetrii.
  • Token usage má být účtovaný přes jasný model: platform subscription, LLM/token usage, konektory, operations a change requesty.
  • Customer-cloud neznamená předání softwaru; má být pozicovaný jako managed private instance.
  • O2 LLM/API klíče mají zůstat v backendu nebo gateway konfiguraci, ne viditelné zákaznickému frontendu nebo tenant uživatelům.
  • Pokud zákazník používá vlastní model subscription, komerční model musí oddělit platform fee od zákazníkem placené modelové spotřeby.
