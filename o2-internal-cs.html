Nákladová struktura O2

Nákladový model je složen ve čtyřech vrstvách: jednorázová implementace, pravidelný provoz, variabilní spotřeba a interní governance. 

Jednorázová implementace

  • Solution design, architektura, bezpečnostní návrh
  • Konfigurace runtime logiky, prompt patterns, guardrails
  • Integrace na API, tool adaptery, SSO/IAM
  • Ingestion pipeline a příprava knowledge zdrojů
  • UAT, hardening, go-live readiness

Provozní service (měsíčně)

  • Monitoring, incident management, L2 podpora
  • Patchování backendu a pravidelná aktualizace komponent
  • Prompt tuning a průběžná optimalizace odpovědí
  • Regresní testy a quality governance
  • Reporting SLA, kvality a hodnoty

Variabilní spotřeba

  • LLM tokeny (input/output), embeddings, reranking
  • AI Search / vector DB dotazy a indexace
  • Storage, compute, síťový přenos, API volání
  • Model fallbacky a špičky zatížení
  • Volání externích nástrojů třetích stran

Interní governance

  • Pre-sales podpora, solutioning, architektonické review
  • Právní a compliance šablony, revize smluv
  • Produktový management, roadmapa, release management
  • Enablement delivery týmu a provozních rolí

Na čem O2 vydělává 

Stabilní model kombinuje setup, měsíční managed službu, variabilní spotřebu a změnové práce.

1) Setup fee (jednorázově)

  • Přeúčtování implementace podle rozsahu a složitosti
  • Oddělit core setup od volitelných custom integrací
  • Změnové požadavky po scope locku účtovat zvlášť

2) Managed service fee (měsíčně) + doplnit subscriptions

  • Provoz, monitoring, podpora, pravidelný tuning
  • SLA tiering: Basic / Business / Enterprise
  • Základní cena musí pokrýt fixní O2 tým

3) Přeprodej cloud services

  • Transparentní pass-through cloudových nákladů


4) Upgrades a change requests

  • Nové funkcionality, nové use cases, nové integrace
  • Upgrade backendu, konektorů, runtime logiky, testů
  • Balíčky změn jako opakovatelná nabídka

5) Add-ons - customizovaný vývoj

  • SARA/testing rozšíření, advanced audit a compliance
  • Voice, widget a analytické dashboardy 
  • Admin tool

6) Advisory a konzultace

  • Conversation design, governance setup, operating model
  • Produktová strategie, roadmapa a kvartální review
  • Školení business a provozních týmů zákazníka

Zakladni HR Buddy baseline tedy pocita s jednorazovou implementaci cca 195 000 Kc a s beznym provozem cca 57 400 Kc za mesic bez admin portalu a bez pripravy dat zakaznika.

Zakladni rozsah reseni

  • Frontend chat jako webovy chat pro zamestnance
  • DB pro konverzace a storage pro dokumenty
  • Azure AI Search: index, indexer a skillset
  • Backend RAG pipeline pro retrieval z KB a odpovedi s odkazem na zdroj
  • AI/LLM gateway pro routing dotazu, logovani spotreby a rate limiting

Co je v mesicnim servisu

  • Monitoring a provoz: dohled, alerting, incident response a sprava zaloh
  • Aktualizace: upgrade BE a FE, AI/LLM gateway a bezpecnostni zaplaty
  • L2 podpora: technicka podpora, konzultace a provozni bug fixing
  • Prompting a prubezna aktualizace knowledge podle provoznich potreb
  • Spotreba infra a tokenu se drzi oddelene jako usage polozka

Kriticke predpoklady a otevrene body

  • MS Azure je provozovano v prostredi O2
  • Je potreba mit pravni sablony: DPA, SLA a smluvni vzory
  • Investice do produktizace musi probehnout pred prvnim prodejem
  • Do pricingu je potreba dodelat licencni model, SLA garance a GDPR role
  • Je potreba rozhodnout, zda je O2 Cloud soucasti standardni nabidky

Co neni v baseline a navysuje cenu

  • Admin portal pro spravu promptu a upload dokumentu do KB
  • Dopracovani admin portalu cca 30 MD
  • Priprava dat zakaznika orientacne 3 az 10 MD podle stavu podkladu
  • Kvalita dat primo ovlivnuje kvalitu odpovedi a rychlost implementace
  • Bez admin portalu je produkt obchodne velmi obtizne prodejny
Nad ramec baseline umime
  • Major upgrade platformy a vyvoj custom integraci
  • Vetsi restrukturalizaci knowledge base
  • Skoleni novych uzivatelu a administratoru
  • Teams nebo Slack integraci podle ciloveho prostredi
  • Rizeni prav k informacim
  • Automaticke revize dokumentace
  • Dotazeni automatizace testovani (kontrola odpovedi modelem vyssiho radu)
Co ma byt v nabidce oddelene
  • Implementace, mesicni servis, Azure infrastruktura a tokenova spotreba
  • Admin portal jako samostatny add-on
  • Priprava a cisteni zakaznickych dat jako samostatna pracnost
  • Repurposing do dalsich domen jako samostatny rozsah

Doporučený komerční model pro O2

Model účtování
  • Setup fee: fixní + scope-based část
  • Měsíční managed fee podle SLA tieru
  • Spotřeba cloudu jako pass-through + markup
  • CR a nové funkce jako samostatný backlog
Smluvní guardraily
  • Výslovný scope lock a change management proces
  • Token budget cap + mechanismus schválení navýšení
  • SLA matice a jasná hranice odpovědností
  • Definice, co je bug fix vs. co je placená změna
Interní řízení profitability
  • Každý deal mít contribution margin dashboard
  • Měsíčně porovnávat plán vs. skutečnost (MD, tokeny, incidenty)
  • Včas eskalovat dealy pod cílovou marží
  • U enterprise zákazníků řídit kvartální repricing
Kdy dává smysl add-on upsell
  • Zákazník požaduje interní správu promptů a knowledge
  • Potřebuje vyšší audit, testování nebo compliance dohled
  • Rozšiřuje se počet use cases nebo kanálů
  • Je potřeba napojení na nové backendové akce

Interní checklist před obchodní nabídkou

1. Je oddělená implementace, service a spotřeba?

Bez tohoto oddělení nelze spolehlivě řídit marži ani srovnávat dealy mezi sebou.

2. Máme token budget cap a alerting?

Spotřeba modelů musí mít limit a mechanismus schválení navýšení na straně zákazníka.

3. Je jasná hranice bug fix vs. change request?

Nejasná hranice vytváří neplacenou práci a rychle snižuje ziskovost provozu.

4. Je v nabídce prostor pro upgrades a roadmapu?

Bez plánovaných upgradů se produkt mění na jednorázový projekt bez opakovatelného výnosu.
